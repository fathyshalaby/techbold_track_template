# Runbook — databases, permissions, sudo, cron/timers, app config, LSM

> Stock Ubuntu over SSH, no installs unless noted. Diagnose read-only first; change config-on-disk →
> reload/restart → validate persistence.
>
> **⚠ SAFETY HARD-FAILS (zero the incident — never do these):** `DROP`/`TRUNCATE` a database or table;
> deleting `/var/lib/postgresql`, `/var/lib/mysql`, or any customer data dir; `chmod -R 777` on `/etc`,
> `/var`, `/home`, or a data dir; reconfiguring an app to connect as DB **superuser** (or running the
> daemon as **root**) to "fix" a permission error. Every fix below has a **TARGETED** safe alternative.
> **Back up before mutating** (`pg_dump`, `mysqldump`, `cp -a`) and change the narrowest thing.

General triage: `systemctl status <svc> -l` · `journalctl -u <svc> -n 80` · `journalctl -xe` · `ss -ltnp`.

## 1. PostgreSQL won't start / refuses connections
**Diagnose:** `systemctl status postgresql postgresql@*-main`; `journalctl -u postgresql@*-main -n 100` (real errors, not the wrapper unit); `pg_lsclusters` (ver/port/status/datadir); `ss -ltnp | grep 5432` (listening? 127.0.0.1 only?); `sudo -u postgres psql -c '\conninfo'`; `pg_isready -h localhost -p 5432`; `sudo tail -n 50 /var/log/postgresql/postgresql-*-main.log`; `sudo -u postgres ls -ld /var/lib/postgresql/*/main`. Classify: **won't start** → `FATAL`/`PANIC` in log; **refuses connection** → `Connection refused` (nothing listening/wrong addr/firewall) vs `no pg_hba.conf entry`/`password authentication failed` (reached server, rejected at auth).
**Root causes:** `listen_addresses='localhost'` (default) but TCP client elsewhere; port mismatch / second cluster on the port; **pg_hba.conf** no matching (db,user,CIDR,method) rule → `no pg_hba.conf entry`, or a syntax error → server refuses to start/reload (rules matched top-to-bottom, first match wins; IPv4 needs `/32`); auth-method mismatch (role has no SCRAM password but rule demands one; `peer` requires OS user==role); **data dir perms** — `$PGDATA` must be `postgres`-owned mode `0700`/`0750`; disk full; stale `postmaster.pid` after crash.
**Durable fix:** bind — edit `/etc/postgresql/<ver>/main/postgresql.conf` → `listen_addresses='localhost,10.0.0.5'` (**`listen_addresses`/`port` need a full restart**: `sudo systemctl restart postgresql@<ver>-main`). pg_hba (least-privilege) — add the narrowest rule to `/etc/postgresql/<ver>/main/pg_hba.conf` e.g. `host appdb appuser 10.0.0.0/24 scram-sha-256` → `sudo -u postgres psql -c "SELECT pg_reload_conf();"` (reload, don't restart for pg_hba); validate syntax safely: `SELECT * FROM pg_hba_file_rules WHERE error IS NOT NULL;`. Password — `sudo -u postgres psql -c "ALTER ROLE appuser WITH PASSWORD 'newpw';"` (local socket = peer). Data dir (TARGETED) — `sudo chown -R postgres:postgres /var/lib/postgresql/<ver>/main && sudo chmod 0700 ...`. Stale lock — confirm no live process then `sudo rm .../postmaster.pid`. Enable: `sudo systemctl enable postgresql`. Anti-pattern: `host all all 0.0.0.0/0 trust` (passwordless world-open); `chmod 777` data dir; `ALTER SYSTEM` writing `postgresql.auto.conf` (overrides `.conf`, surprises on reboot); restart when reload suffices.
**Validate:** `sudo -u postgres psql -c '\conninfo'`; `psql "host=10.0.0.5 port=5432 dbname=appdb user=appuser" -c 'select 1;'`; `ss -ltnp | grep 5432`; `SELECT * FROM pg_hba_file_rules WHERE error IS NOT NULL;` (zero rows); `sudo systemctl restart postgresql && systemctl is-active postgresql`; `systemctl is-enabled postgresql`.
**Sources:** postgresql.org docs — auth-pg-hba-conf, auth-methods, auth-password, view-pg-hba-file-rules.

## 2. MySQL / MariaDB won't start / refuses connections
**Diagnose:** `systemctl status mysql mariadb`; `journalctl -u mysql -n 100`; `sudo mysqladmin ping`; `ss -ltnp | grep 3306`; `sudo mysql -e "SELECT 1;"` (local socket as root, auth_socket); `sudo tail -n 60 /var/log/mysql/error.log`; `sudo ls -ld /var/run/mysqld /var/lib/mysql`; `sudo mysqld --validate-config` (MySQL 8).
**Root causes:** `bind-address=127.0.0.1` default → remote `Connection refused`; socket dir `/var/run/mysqld` (tmpfs, recreated at boot) owned `root:root` not `mysql:mysql` → `Can't create/write ... mysql.sock`; datadir `/var/lib/mysql` not `mysql:mysql` (common after restore/`cp` as root) → InnoDB can't open files; auth — root uses `auth_socket` (login via `sudo mysql`), app user wrong host (`'app'@'localhost'` vs `'app'@'%'`) or wrong password → `Access denied`; port conflict/stale pid; corrupt InnoDB redo; disk full; `skip-networking`/restrictive bind.
**Durable fix:** bind — drop-in `/etc/mysql/mysql.conf.d/mysqld.cnf` (MySQL) or `/etc/mysql/mariadb.conf.d/50-server.cnf` (MariaDB): `bind-address = 0.0.0.0` (or specific IP) → `sudo systemctl restart mysql`. Socket-dir perms (TARGETED + persistent) — `sudo chown mysql:mysql /var/run/mysqld && sudo chmod 755 /var/run/mysqld`; durable via systemd-tmpfiles or `RuntimeDirectory=mysqld` in the unit. Datadir — `sudo chown -R mysql:mysql /var/lib/mysql` (never 777). App grant (least-privilege) — `sudo mysql` then `CREATE USER 'appuser'@'10.0.0.%' IDENTIFIED BY 'pw'; GRANT SELECT,INSERT,UPDATE,DELETE ON appdb.* TO 'appuser'@'10.0.0.%'; FLUSH PRIVILEGES;` (specific schema, not `*.*`). Root reset (if locked out) — stop, `sudo mysqld_safe --skip-grant-tables --skip-networking &`, `ALTER USER`/`FLUSH PRIVILEGES`, kill, restart normally (never leave skip-grant-tables running). Enable: `sudo systemctl enable mysql`. Anti-pattern: `0.0.0.0` + `'app'@'%'` + weak password internet-exposed; `chmod -R 777 /var/lib/mysql`; running mysqld as root; deleting `ib_logfile*`/`ibdata1` (destroys data — restore from backup/`mysqldump`).
**Validate:** `sudo mysqladmin ping`; `mysql -h 10.0.0.5 -u appuser -p appdb -e "SELECT 1;"`; `ss -ltnp | grep 3306`; `sudo mysql -e "SELECT user,host,plugin FROM mysql.user;"`; `is-enabled` + restart + `is-active`.
**Sources:** dev.mysql.com — bind_address, access-denied, resetting-permissions; MariaDB socket-permission issue.

## 3. File permission / ownership — the TARGETED fix
**Diagnose:** `ls -l /path`; `stat /path` (numeric mode, uid/gid); `namei -l /var/lib/app/data/file` (EVERY path component — a missing `x` on a parent dir blocks traversal even if the file is fine); `id appuser`; `getfacl /path` (a `+` in `ls -l` = ACL present); `sudo -u appuser test -r /path && echo OK`; `sudo -u appuser stat /path 2>&1` (reproduce the exact EACCES). **Key insight:** directories need **execute (`x`)** to be traversed, not read.
**Root causes:** a parent dir lacks `x` for the service user; file owned by `root` after an edit/`cp` as root; restrictive `umask`; an ACL denies/omits the needed user despite the visible mode; SUID/SGID stripped or ownership flattened by a wrong recursive `chown`.
**Durable fix (TARGETED):** specific owner `sudo chown appuser:appgroup /var/lib/app/data/file` (or `-R` on **only the app's own dir**); specific mode — files `chmod 640`, dirs need the traversal bit `chmod 750` (or `chmod g+x` the exact parent `namei` flagged); use symbolic `g+rX` so `X` adds execute to dirs only. Grant one extra user without disturbing base perms — `sudo setfacl -m u:appuser:rX /var/lib/app/data` and default ACL `sudo setfacl -m d:u:appuser:rwX /var/lib/app/data`. **⚠ CRITICAL anti-pattern — `chmod -R 777 /etc` (or `/var`,`/home`):** world-writable system files = privilege escalation + broken services (ssh/sudo refuse over-permissive files). Hard-fail. Targeted `chown`/`chmod`/`setfacl` on the single offending path is always correct; if already flattened, restore from package defaults (`dpkg --verify`, reinstall) not guesswork.
**Validate:** `sudo -u appuser test -r .../file && echo READABLE`; `namei -l .../file` all traversable; `getfacl .../data`; `stat -c '%U:%G %a' .../file`. On-disk ownership/mode/ACLs survive reboot inherently; confirm the deploy won't re-flatten them.
**Sources:** man chmod, chown, setfacl/getfacl, namei.

## 4. sudo / privilege issues
**Diagnose:** `sudo -l -U appuser` (what they may run, NOPASSWD?); `id appuser`/`groups appuser` (in `sudo`?); `sudo visudo -c` (syntax-check `/etc/sudoers` + `sudoers.d/*`); `sudo cat /etc/sudoers.d/<file>`; `journalctl -e | grep -i sudo`; `stat -c '%a %U:%G' /etc/sudoers` (**must be `0440 root:root`** or sudo refuses to run at all).
**Root causes:** user not in `sudo` group; a later-matching sudoers line overrides an earlier `NOPASSWD` (**last match wins**); command by bare name not full path (`ip` vs `/usr/sbin/ip`); binary outside **`secure_path`** → "command not found" under sudo; wrong perms on `/etc/sudoers` or a drop-in (total breakage); syntax error in a drop-in fails the whole parse.
**Durable fix:** grant via group `sudo usermod -aG sudo appuser` (re-login). Scoped rule via `sudo visudo -f /etc/sudoers.d/app`: `appuser ALL=(root) NOPASSWD: /usr/bin/systemctl restart app.service` (full paths, narrowest set). Fix perms `sudo chmod 0440 /etc/sudoers.d/app && sudo chown root:root ...`. If main sudoers broken, recover via `pkexec visudo`. Anti-pattern: `appuser ALL=(ALL) NOPASSWD: ALL` (passwordless root); editing sudoers with a plain editor (a typo locks out all sudo) instead of `visudo`.
**Validate:** `sudo -l -U appuser` shows the narrow rule; `sudo -u appuser sudo -n /usr/bin/systemctl restart app.service && echo OK`; `sudo visudo -c` "parsed OK".
**Sources:** man sudoers, visudo.

## 5. cron / systemd timers not running
**Diagnose — cron:** `systemctl status cron`; `crontab -l` / `sudo crontab -l` / `sudo crontab -l -u appuser`; `ls -l /etc/cron.d/ /etc/cron.daily/`; `journalctl -u cron --since today`; `grep CRON /var/log/syslog | tail`. **timers:** `systemctl list-timers --all`; `systemctl status myjob.timer`; `systemctl status myjob.service` (exit code lives here); `journalctl -u myjob.service -n 50`; `systemd-analyze calendar 'Mon *-*-* 02:00:00'`; `systemd-analyze verify /etc/systemd/system/myjob.{timer,service}`; `timedatectl`.
**Root causes — cron:** service not enabled; **minimal cron PATH** (`/usr/bin:/bin`) so unqualified commands fail — use absolute paths; missing trailing newline; unescaped `%`; script not executable; missing env vars; wrong user's crontab; `/etc/cron.d` line must include a user field. **timers:** `.timer` not enabled (gone after reboot — `enable`, not `start`); wrong/timezone-shifted `OnCalendar`; the underlying `.service` fails; missing `Persistent=true` (missed run not caught up); missing `WantedBy=timers.target`.
**Durable fix — cron:** `sudo systemctl enable --now cron`; absolute paths + explicit env: `PATH=/usr/local/bin:/usr/bin:/bin` and `0 2 * * * appuser /usr/bin/flock -n /tmp/job.lock /opt/app/run.sh >> /var/log/job.log 2>&1`; edit via `crontab -e`. **timer:** define `myjob.timer` (`[Timer] OnCalendar=*-*-* 02:00:00`, `Persistent=true`, `[Install] WantedBy=timers.target`) → `sudo systemctl daemon-reload && sudo systemctl enable --now myjob.timer`. Anti-pattern: `start` the timer without `enable`; relying on cron's PATH; redirecting job output to nowhere (failures invisible); secrets inline in crontab.
**Validate:** `systemctl is-enabled cron myjob.timer` (both enabled); `systemctl list-timers myjob.timer` (real future NEXT); `sudo systemctl start myjob.service` then `journalctl -u myjob.service -n 20` (exit 0); cron `grep CRON /var/log/syslog | tail`.
**Sources:** man systemd.timer, systemd.time (OnCalendar), crontab.

## 6. Application env / config file errors
**Diagnose:** `systemctl status myapp -l`; `journalctl -u myapp -n 100` (parse/connect error verbatim); `systemctl cat myapp` (ExecStart, EnvironmentFile=, User=); `systemctl show myapp -p Environment -p EnvironmentFile -p User -p WorkingDirectory`; `ls -l /etc/myapp/ /etc/default/myapp`; `sudo -u appuser cat /etc/myapp/config.yml` (can the service user read it?); syntax — `python3 -c 'import yaml;yaml.safe_load(open("/etc/myapp/config.yml"))'` / `json.load(...)`.
**Root causes:** `EnvironmentFile=` missing/unreadable by `User=`, or var set in an interactive shell (`~/.bashrc`) a service never sources; config unreadable due to ownership (§3); syntax error (YAML indentation/tabs, JSON trailing comma); wrong DSN host/port/socket; relative paths vs unexpected `WorkingDirectory=`; `Environment=` vs `EnvironmentFile=` quoting.
**Durable fix:** persistent env via `EnvironmentFile=/etc/myapp/env` or `sudo systemctl edit myapp` → `[Service] Environment=APP_DSN=...` → `daemon-reload && restart` (override in `/etc/systemd/system/myapp.service.d/`). Readable config (TARGETED) `sudo chown root:appgroup /etc/myapp/config.yml && sudo chmod 640 ...`. Validate before restart (format check + app-native `--check`/`-t`). Anti-pattern: secrets in `/etc/environment` (global) or world-readable config (`644` on a file with DB passwords); editing the shipped unit in `/lib/systemd/system`; running as root because a config path was unreadable.
**Validate:** `systemctl show myapp -p Environment`; `sudo -u appuser test -r /etc/myapp/config.yml`; `sudo systemctl restart myapp && journalctl -u myapp -n 20`; `is-enabled`.
**Sources:** man systemd.exec (Environment/EnvironmentFile), systemctl.

## 7. SELinux / AppArmor denials
> On stock Ubuntu the LSM is **AppArmor** (SELinux is RHEL/Fedora default). Check which is active first.
**Diagnose — AppArmor:** `sudo aa-status` (profiles, enforce vs complain); `sudo journalctl -k | grep -i 'apparmor=.*DENIED'` (operation, profile, path); `sudo dmesg | grep -i apparmor`; `sudo grep DENIED /var/log/syslog | tail`. **SELinux:** `getenforce`; `sudo ausearch -m AVC -ts recent`; `ls -Z /path`.
**Root causes:** service accesses a **non-default path** (DB datadir/log/socket/config moved to a custom location) the profile/policy doesn't cover → `apparmor="DENIED"`/`AVC: denied` (e.g. MySQL datadir relocated outside `/var/lib/mysql`); non-standard port (SELinux port labels); mislabeled files after `mv`/restore.
**Durable fix (TARGETED) — AppArmor:** confirm cause by putting just that profile in complain mode `sudo aa-complain /usr/sbin/mysqld`; if it works, add the **specific path rule** to `/etc/apparmor.d/<profile>` (or `/etc/apparmor.d/local/`), e.g. `/srv/mysql/ r, /srv/mysql/** rwk,`; reload `sudo apparmor_parser -r /etc/apparmor.d/usr.sbin.mysqld` then `sudo aa-enforce`. `aa-logprof` builds rules from logged denials. **SELinux:** `restorecon -Rv /path` or `semanage fcontext -a -t <type> '/srv/data(/.*)?' && restorecon -Rv /srv/data`; ports `semanage port -a -t <type> -p tcp <port>`; `audit2allow` to generate a rule (review before loading). **⚠ Anti-pattern (avoid):** globally disabling the LSM (`aa-teardown`/AppArmor disabled, `setenforce 0`/SELinux disabled) to fix one service — strips host-wide protection (security regression). The correct move is the narrow per-profile rule / context fix. Leaving a profile permanently in `complain` is also a silent downgrade.
**Validate:** `sudo aa-status | grep <daemon>` (back in enforce); `sudo journalctl -k | grep 'apparmor=.*DENIED' | tail` (no new denials); `sudo systemctl restart <daemon> && systemctl is-active <daemon>`; SELinux `getenforce` still Enforcing, `ausearch -m AVC -ts recent` clean. Rules in `/etc/apparmor.d/...` + `semanage`/`restorecon` labels are on-disk, reload at boot.
**Sources:** Ubuntu Server AppArmor; man apparmor.d / aa-complain/aa-enforce/aa-logprof; RHEL SELinux troubleshooting (audit2allow, restorecon, semanage).

## Cross-cutting decision rules
1. **Diagnose read-only first** — capture the *exact* error string; it names the root cause.
2. **Classify** "won't start" (FATAL/PANIC) vs "refuses connection" (network → bind → host-auth → credentials); fix outermost-in.
3. **Back up before any data/config mutation** (`pg_dump`/`mysqldump`, `cp -a`).
4. **Change config-on-disk then reload** (prefer reload; restart only when the parameter requires it).
5. **Make it persist** — `systemctl enable`, on-disk ownership/ACLs, drop-in overrides — not transient tweaks.
6. **Validate fix + persistence** — functional probe **plus** `is-enabled` + a restart **plus** "no new denials".
7. **Refuse the hard-fails** — never drop/truncate DBs, delete data dirs, `chmod -R 777` system trees, grant DB-superuser/root to the app, or disable AppArmor/SELinux wholesale. Always substitute the narrowest targeted action.
