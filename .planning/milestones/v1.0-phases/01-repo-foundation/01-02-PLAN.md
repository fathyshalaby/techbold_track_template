<objective>
Implement the Zod-validated env parser and the mock-flag resolver using TDD.

These are the two pure functions at the center of Phase 1:
- env.ts parses process.env through a Zod schema, fails fast with a readable message naming the
  missing variable when any required key is absent, and exports a typed config object.
- resolveClientMode(service) implements the D-01 flag precedence: MOCK_MODE=true forces all
  services to mock; when false, the per-service flag (MOCK_PHOENIX / MOCK_SSH / MOCK_LLM)
  decides independently.

These functions have fully defined I/O — ideal TDD candidates. The health route (Plan 03) reads
isMockMode() to populate the mode field; later phases call resolveClientMode() to select real vs
mock clients.

Purpose: Establish the config contract all downstream code depends on before any route or
client code is written.

Output: backend/src/env.ts (real implementation), backend/src/tests/env.test.ts (full suite).
</objective>

<feature>
  <name>env parser + mock-flag resolver</name>
  <files>backend/src/env.ts, backend/src/tests/env.test.ts</files>

  <behavior>
    Env parser (Zod schema + fail-fast):
    - Required string: PHOENIX_API_URL — missing → process.exit(1) with message:
      "Missing required env var: PHOENIX_API_URL"
    - Required string: PHOENIX_API_TOKEN — missing → exit with named var in message
    - Required string: OPENAI_API_KEY — missing → exit with named var in message
    - Optional with default: LLM_PROVIDER (default "openai")
    - Optional with default: LLM_MODEL (default "gpt-4o")
    - Optional with default: SSH_KEY_PATH (default "/keys/id_rsa")
    - Optional boolean (default false): MOCK_MODE, MOCK_PHOENIX, MOCK_SSH, MOCK_LLM
    - Boolean env vars: "true" string → true; anything else → false

    Mock-flag resolver (resolveClientMode):
    - Signature: resolveClientMode(service: 'phoenix' | 'ssh' | 'llm') → 'mock' | 'real'
    - Rule: if env.MOCK_MODE is true → return 'mock' regardless of service
    - Rule: if env.MOCK_MODE is false and env.MOCK_PHOENIX is true and service === 'phoenix' → 'mock'
    - Rule: if env.MOCK_MODE is false and env.MOCK_SSH is true and service === 'ssh' → 'mock'
    - Rule: if env.MOCK_MODE is false and env.MOCK_LLM is true and service === 'llm' → 'mock'
    - Rule: all other cases → 'real'

    isMockMode convenience helper:
    - Returns true when ANY service resolves to 'mock' (i.e., MOCK_MODE=true OR at least one
      per-service flag is true). Used by health route for mode field.
    - Simpler alternative accepted: isMockMode() returns env.MOCK_MODE directly. Planner leaves
      exact semantics to executor's judgment, as long as health route correctly shows 'mock'
      when MOCK_MODE=true.

    Test cases (env.test.ts):
    - parseEnv with all required vars present → returns typed object with correct values
    - parseEnv with PHOENIX_API_URL missing → throws/exits with message containing "PHOENIX_API_URL"
    - parseEnv with PHOENIX_API_TOKEN missing → message contains "PHOENIX_API_TOKEN"
    - parseEnv with OPENAI_API_KEY missing → message contains "OPENAI_API_KEY"
    - parseEnv MOCK_MODE "true" string → env.MOCK_MODE === true
    - parseEnv MOCK_MODE absent → env.MOCK_MODE === false (default)
    - resolveClientMode('phoenix') when MOCK_MODE=true → 'mock'
    - resolveClientMode('ssh') when MOCK_MODE=true → 'mock'
    - resolveClientMode('llm') when MOCK_MODE=true → 'mock'
    - resolveClientMode('phoenix') when MOCK_MODE=false, MOCK_PHOENIX=true → 'mock'
    - resolveClientMode('ssh') when MOCK_MODE=false, MOCK_PHOENIX=true → 'real'
    - resolveClientMode('llm') when MOCK_MODE=false, MOCK_LLM=true → 'mock'
    - resolveClientMode('phoenix') when MOCK_MODE=false, MOCK_PHOENIX=false → 'real'
    - All three services → 'real' when all mock flags false
  </behavior>

  <implementation>
    The test file must be created and run (RED — all fail) before env.ts is implemented.

    For testing the fail-fast behavior without actually calling process.exit(1) in tests:
    structure env.ts so parseEnv(rawEnv: Record string string) is a pure exported function that
    throws a typed error (or returns a result type) when validation fails. The module-level
    export const env = parseEnv(process.env) is what calls it on startup. Tests call parseEnv()
    directly with a fabricated env object — they never hit the module-level side effect.

    Zod schema approach: z.object({ ... }).parse() wrapped in a try/catch that calls
    console.error("Missing required env var: " + missingField) and process.exit(1).
    Alternatively, use z.object().safeParse() and check success. Either pattern is fine as
    long as tests can reach the validation logic directly.

    resolveClientMode is a pure function; it receives no arguments beyond the service name and
    reads from the module-level env. Tests must be able to control env values — use dependency
    injection (pass env as second argument with a default) or re-export a testable overload.

    After GREEN, refactor for clarity: separate the Zod schema definition from the parse call,
    export the schema type (EnvConfig) for downstream use.

    Boolean coercion: use z.preprocess(v => v === 'true', z.boolean()) or z.coerce.boolean()
    to handle the string-to-boolean conversion from process.env.
  </implementation>
</feature>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| process.env → env.ts | All secrets enter here; env.ts is the single chokepoint |
| env.ts → callers | Callers receive typed values, never raw process.env strings |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | env.ts error messages | mitigate | Error messages must name the missing variable key only — never print the value of any env var in error output. Enforced by test: assert error message contains key name, does not contain token value. |
| T-02-02 | Tampering | process.env mutation in tests | accept | Tests fabricate env objects passed to parseEnv() directly; they never mutate process.env. No global state pollution between test cases. |
| T-02-03 | Denial of Service | fail-fast exit on missing var | accept | Intentional behavior per PLAT-02 and ARCHITECTURE.md §10. Silent misconfiguration with a default is more dangerous than a loud exit. |
| T-02-SC | Tampering | npm/pip/cargo installs | accept | No new installs in this plan — all deps defined in Plan 01's package.json. Vitest and Zod are already declared. |
</threat_model>
