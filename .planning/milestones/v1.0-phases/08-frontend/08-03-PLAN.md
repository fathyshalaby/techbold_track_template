    Note: the prop is onBack (not onDone). Plan 08-05 implements this interface exactly.

    Replace frontend/src/App.tsx with the view-state shell:

    State:
    - currentView: 'list' | 'run' | 'activity' — useState default 'list'
    - activeRunId: string | null — useState default null
    - activeTicketTitle: string — useState default ''
    - customerSystem: CustomerSystem | null — useState default null
    - activityDraft: ActivityDraft | null — useState default null
    - createError: string | null — useState default null
    - creating: boolean — useState default false
    - draftingActivity: boolean — useState default false

    The shell renders:
    - When currentView === 'list': TicketListView with an onSelectTicket callback.
    - When currentView === 'run' and activeRunId is non-null: RunView with runId,
      ticketTitle, customerSystem, and onActivityReady callback (see below).
    - When currentView === 'activity' and activeRunId is non-null: ActivityView with
      runId, activityDraft, and onBack callback that sets currentView back to 'run'
      (not 'list') so the technician can continue the run if needed.

    onSelectTicket(ticket: Ticket):
    - Sets creating to true, clears createError.
    - Calls createRun(ticket.id) from api.ts.
    - On success: sets activeRunId to response.runId, sets activeTicketTitle to
      ticket.title, sets customerSystem to response.customerSystem, sets
      currentView to 'run'.
    - On failure: sets createError to the error message string.
    - Always sets creating to false in finally.

    onActivityReady() — called by RunView when the run reaches WAITING_FOR_ACTIVITY_REVIEW:
    - Sets draftingActivity to true.
    - Calls draftActivity(activeRunId!) from api.ts.
    - On success: sets activityDraft to the returned ActivityDraft, sets
      currentView to 'activity'.
    - On failure: sets createError to the error message (draft failed — user sees
      banner and stays on run view; they can retry by clicking the activity button again).
    - Always sets draftingActivity to false.

    If createError is set, render it as a small error banner above the current view
    (class "error-banner") that the user can dismiss (button sets createError null).

    If draftingActivity is true, render a non-blocking overlay or text "Drafting
    activity report…" (class "drafting-overlay") so the technician knows work is
    happening. This must NOT block the abort button in RunView.

    A thin header bar renders "AI Service Desk Autopilot" as an h1 and, when
    currentView !== 'list', a "← Back to tickets" button that sets currentView to
    'list' and clears activeRunId and customerSystem.

    No router library. No CSS framework. Use class names for styling hooks; the CSS
    is filled in by Task 2.
