<objective>
Implement the durable run store — the append-only audit log + run lifecycle CRUD that every later
phase writes to. This is what the judge inspects for rubric C and what the activity generator
reads for rubric B. No HTTP routes, no agent calls — pure data layer.

Purpose: Every run event, command decision, SSH result, and observation must survive a process
restart. SQLite is the primary store; JSONL is the fallback so the demo survives a native-build
failure in the hackathon environment.

Output: Four modules (schema, db, runs, audit) plus the `ulid` dependency added to package.json.
</objective>

  <action>
    Export `StoreMode` type: `'sqlite' | 'jsonl'`.

    Export `DbAdapter` type with the interface the rest of the store layer uses:
    - `run(sql: string, params?: unknown[]): void` — for INSERT/UPDATE
    - `get<T>(sql: string, params?: unknown[]): T | undefined` — for single-row SELECT
    - `all<T>(sql: string, params?: unknown[]): T[]` — for multi-row SELECT
    - `mode: StoreMode`

    Export `getDb(dbPath?: string): DbAdapter`. Implement lazy singleton init:
    1. On first call, attempt `await import('better-sqlite3')` (dynamic import so the module loads even when native bindings are absent at the module-load phase — detect failure in the catch, not at top-level).
    2. On success: open the SQLite file at `dbPath ?? ':memory:'` (`:memory:` default for tests; real path supplied by the caller in production). Run `db.pragma('journal_mode = WAL')`. Execute the 6 CREATE TABLE IF NOT EXISTS statements matching ARCHITECTURE §6 column definitions exactly. Set `mode: 'sqlite'`.
    3. On failure (native binding load error): activate JSONL fallback. Log `[store] SQLite unavailable — using JSONL fallback` once via `console.warn`. JSONL adapter:
       - Maintains an in-memory `Map<string, unknown[]>` per table name.
       - `run(sql, params)`: parse the table name from the INSERT statement, push params as a record. Supports only INSERT operations (all the store needs); UPDATEs are approximated by replacing the matching record by id in the array.
       - `get<T>`: parse WHERE id = ? from the SQL, find by id in the map.
       - `all<T>`: return all records for the table.
       - `mode: 'jsonl'` — the name is "JSONL" per the architecture doc (in a real impl the fallback would write `.jsonl` files; for the hackathon the in-memory map is sufficient and the name is preserved for compatibility).
    4. Cache the adapter in a module-level variable so subsequent `getDb()` calls return the same instance.

    No top-level side effects (no `getDb()` call at import time). ESM `.js` suffixes.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → appendAuditEvent | Caller must pass already-redacted payload; the store does not re-redact |
| store → SQLite file | DB file on disk; no auth — single-machine local tool, acceptable per ARCHITECTURE §10 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-09 | Tampering | audit_events delete path | mitigate | No delete/update method exported from audit.ts — enforced structurally by API absence, not a soft-delete column or convention |
| T-03-10 | Information Disclosure | unredacted payload in audit write | mitigate | appendAuditEvent contract: caller passes redacted payload; orchestrator (Phase 5) must call redactSecrets before constructing the payload object |
| T-03-11 | Tampering | ULID collision | accept | ULID is time-ordered + 80-bit random component; collision probability negligible for single-process single-run tool |
| T-03-12 | Denial of Service | unbounded JSONL fallback memory | accept | Fallback is for demo/hackathon environment; a single incident run produces O(hundreds) of records; memory risk is negligible for the use case |
| T-03-13 | Elevation of Privilege | SQLite file permissions | accept | Single-team local tool; no multi-tenant access; file lives inside the Docker container or local dev directory |
| T-03-SC | Tampering | npm installs | mitigate | `ulid` is a well-known zero-dependency ULID package on npmjs.com; verify on install with slopcheck |
</threat_model>
