<objective>
Implement the deterministic safety core: the blocklist policy (`command-policy.ts`) and the risk
classifier (`classifier.ts`). These are the two functions every command passes through — at
proposal time and again after any human edit (per SAFE-05). This is rubric-C's hardest
correctness requirement and the graders inspect both the audit records AND the source.

Purpose: Guarantee that no dangerous command can reach an SSH executor regardless of how it was
composed or obfuscated. The audit record of a blocked command includes the rule name and reason
so the grader can see the safety reasoning.

Output: Two fully implemented, tested modules exported as a small public API:
`validateCommandAgainstPolicy`, `classifyCommand`, `BLOCKLIST`, and supporting types.
</objective>

    Implement command-policy.ts:

    Export `BlocklistRule` type: `{ pattern: RegExp; ruleName: string; reason: string }`.

    Export `PolicyResult` type: `{ allowed: boolean; riskLevel: RiskLevel; matchedRule?: string; reason?: string }`.

    Export `BLOCKLIST` as a readonly array of `BlocklistRule`. Represent every §3 bullet as one or more labeled rules. Rule categories and their `ruleName` strings (use these exact names — they appear in audit records):
    - `rm-rf-system-paths` — rm -rf on /, /*, ~, and system directories (/etc, /var, /home, /srv, /usr, /boot, /var/lib/*)
    - `recursive-find-delete` — find … -delete over system paths
    - `disk-wipe` — mkfs.*, mke2fs, dd if=…of=/dev/*, wipefs, shred on devices
    - `block-device-write` — direct writes to /dev/sd* or /dev/hd*
    - `shutdown-reboot` — shutdown, reboot, halt, poweroff, init 0, init 6
    - `fork-bomb` — :(){ :|:& };: and variants
    - `broad-chmod-chown` — chmod -R 777 / chmod 777 -R on system trees; chown -R on /
    - `disable-security` — ufw disable, iptables -F, systemctl stop/disable/mask firewalld/ufw/auditd, setenforce 0
    - `secret-exposure` — cat /etc/shadow, cat ~/.ssh/id_*, .env dumps, printenv piped externally
    - `hide-tracks` — history -c, rm/truncate of /var/log/*, journalctl --vacuum, echo > ~/.bash_history
    - `exfiltration` — curl … | sh, wget … | sh, curl/wget POSTing to external, nc/netcat reverse shells
    - `db-destruction` — DROP DATABASE, DROP TABLE, TRUNCATE (SQL keyword — matches `TRUNCATE TABLE` and bare `TRUNCATE`), dropdb, rm of /var/lib/postgresql or /var/lib/mysql, re-initdb
    - `run-as-root` — sudo su root, exec as root to bypass permissions
    - `mass-kill` — kill -9 -1, pkill of system services, kill PID 1

    Normalization function (internal, not exported): `normalizeCommand(cmd: string): string`. Steps:
    1. Trim leading/trailing whitespace.
    2. Collapse runs of whitespace to a single space.
    3. Remove wrapping single or double quotes around individual tokens (strip `'rm'` → `rm`).
    4. Attempt to resolve simple `$(echo foo)`-style wrappers where the inner string is a plain literal — replace with the literal. If the `$()` or backtick content is non-trivial (contains spaces, pipes, variables), mark the command as unresolvable.
    5. If unresolvable, return a sentinel that will match the `fork-bomb` or a catch-all BLOCK rule — simplest approach: if the normalized form still contains `$(` or a backtick with non-trivial content, block it by prepending `__UNRESOLVABLE__ ` (the caller checks for this prefix and returns HIGH_RISK_BLOCKED with rule name `unresolvable-wrapper`).

    Chained command handling: after normalization, split the command string on unescaped `;`, `&&`, and `||` separators to obtain individual segments. Trim each segment. Run the blocklist regexes against each segment independently (no start-anchors on the regexes — they must match anywhere in the segment string). If any segment matches a blocklist rule, return HIGH_RISK_BLOCKED with that rule. This means `echo hi; rm -rf /etc` is caught at the `rm -rf /etc` segment. No regex should use a `^` start-anchor that would prevent matching mid-string in a segment.

    Export `validateCommandAgainstPolicy(command: string): PolicyResult`. Logic:
    1. Normalize the command.
    2. Check for `__UNRESOLVABLE__` prefix → return `{ allowed: false, riskLevel: HIGH_RISK_BLOCKED, matchedRule: 'unresolvable-wrapper', reason: 'Command contains unresolvable subshell or backtick — blocked conservatively' }`.
    3. Split the normalized command into segments on unescaped `;`, `&&`, `||`.
    4. For each segment, test against each rule in `BLOCKLIST` in order. First match on any segment → return `{ allowed: false, riskLevel: HIGH_RISK_BLOCKED, matchedRule: rule.ruleName, reason: rule.reason }`.
    5. If no blocklist match on any segment, call `classifyCommand(command)` and return `{ allowed: true, riskLevel: <result> }`.

    Implement classifier.ts:

    Export `classifyCommand(command: string): RiskLevel`. Deterministic precedence (top to bottom, first match wins):
    1. If `validateCommandAgainstPolicy` would block it (i.e., the blocklist matches) → return `HIGH_RISK_BLOCKED`. To avoid circular import, the classifier calls its own internal blocklist check (re-use the normalize + BLOCKLIST array imported from command-policy). Alternatively, structure it so classifier imports BLOCKLIST from command-policy but command-policy calls classifyCommand — this creates a cycle. Preferred structure: classifier does NOT call validateCommandAgainstPolicy; instead command-policy imports classifyCommand and calls it as step 5 after the blocklist pass. The classifier itself only handles levels SAFE_READ_ONLY / LOW_RISK_CHANGE / MEDIUM_RISK_CHANGE.
    2. Allowlist patterns (SAFE_READ_ONLY — must match tightly to a specific-target, bounded, read-only shape):
       - `systemctl status <svc> [--no-pager]`, `systemctl is-active <svc>`, `systemctl is-enabled <svc>`
       - `journalctl -u <svc>`, `journalctl -p err`
       - `df -h`, `free -m`, `lsblk`, `du -sh <path>`, `uname -a`, `uptime`, `whoami`, `id`, `hostnamectl`
       - `ps aux`, `ss -tulpn`, `ip a`, `ip addr`
       - `ping -c <n> <host>`
       - `curl -I localhost:…`, `curl -s -o /dev/null -w …`
       - `cat <path>` where path does NOT match secret paths (/etc/shadow, ~/.ssh/, .env)
       - `tail -n <n> <path>`, `grep <pattern> <path>`, `ls -la <path>`, `stat <file>`, `getfacl <file>`
    3. LOW_RISK_CHANGE patterns:
       - `systemctl restart <specific-svc>`, `systemctl start <specific-svc>`, `systemctl enable <specific-svc>`
       - `chown <user>[:<group>] <specific-path>` (non-recursive, non-system-root)
       - `chmod <mode> <specific-path>` (non-recursive, non-broad-system-path)
       - `mkdir -p <path>`, `touch <path>`
       - `cp <src> <dst>` (file copy / backup)
       - `mv <src> <dst>`
       - Targeted `sed -i` on a specific file
    4. All remaining recognized commands → MEDIUM_RISK_CHANGE.
    5. Unrecognized command (no rule matched) → MEDIUM_RISK_CHANGE (never silently SAFE_READ_ONLY per SAFE-02).

    File structure rules: use ESM `.js` import suffixes. No top-level side effects. No fenced code in action. Keep the module dependency direction: classifier imports RiskLevel from risk-levels; command-policy imports BOTH RiskLevel from risk-levels AND classifyCommand from classifier. Commit: `feat(03-01): implement blocklist policy and risk classifier`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| agent → policy gate | AI-proposed command string enters validateCommandAgainstPolicy; treated as untrusted |
| technician edit → policy gate | Human-edited command re-enters the same gate; treated as untrusted regardless of source |
| command string → normalizer | Raw string may contain obfuscation attempts before pattern matching |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | normalizeCommand | mitigate | Collapse whitespace + strip obfuscation quotes + sentinel unresolvable wrappers before any regex match; tested with obfuscation variants in safety-policy.test.ts |
| T-03-02 | Elevation of Privilege | classifyCommand fallthrough | mitigate | Unknown commands default to MEDIUM_RISK_CHANGE — never fall through to SAFE_READ_ONLY silently; tested with `someobscurecommand` case |
| T-03-03 | Tampering | edited-command recheck bypass | mitigate | validateCommandAgainstPolicy is stateless pure function; calling it a second time on the edited string is identical to the first call — no bypass path; SAFE-05 tested directly |
| T-03-04 | Information Disclosure | BLOCKLIST rule reason field | accept | Rule names and reasons are internal audit data (not returned to the browser raw); low-value disclosure, high audit benefit |
| T-03-05 | Tampering | chained command bypass via ; && \|\| | mitigate | Normalized string split on unescaped shell separators; each segment validated independently — `echo hi; rm -rf /etc` caught at second segment |
| T-03-SC | Tampering | npm installs | mitigate | No new npm packages in this plan; no install step required |
