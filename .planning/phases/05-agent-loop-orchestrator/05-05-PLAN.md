    GREEN phase: Add the async driver to orchestrator.ts alongside the existing reduce() function.

    advance(runId: string, incomingEvent?: OrchestratorEvent, db?: DbAdapter): Promise&lt;OrchestratorState&gt;

    Driver logic:

    1. Load current run from store via getRunById(runId). Reconstruct OrchestratorState from run row:
       { runId, phase: run.current_phase as RunPhaseValue, status: run.status as RunStatusValue,
         stepCount: read from a meta observation or default 0, ticketId: run.ticket_id,
         customerSystemId: run.customer_system_id }

    2. If incomingEvent provided: call reduce(state, incomingEvent), perform side-effects (see below),
       persist new phase, return nextState.

    3. If no incomingEvent (auto-advance): based on current phase, call the appropriate agent:
       - CREATED → transition to LOADED_CONTEXT (load ticket + customerSystem from Phoenix client),
         then immediately transition to TRIAGING; emit 'run.started' audit event
       - LOADED_CONTEXT → transition to TRIAGING
       - TRIAGING → call runProblemAnalyzer(runId, ticket, observations)
         On success: build diagnostic_proposal_ready event; call validateCommandAgainstPolicy(proposal.command)
           If blocked: call reduce(state, { type: 'command_blocked', reason, command }); perform effects; return
           If allowed: call reduce(state, { type: 'diagnostic_proposal_ready', proposal }); perform effects; return
         On failure: write 'agent.unavailable' audit event; return state unchanged (do not throw)
       - PLANNING_FIX → call runProblemSolver(runId, hypothesis, observations)
         On success: build fix_proposal_ready event; run validateCommandAgainstPolicy on proposal.command
         On failure: same degradation as above
       - VALIDATING → call runValidator(runId, observations)
         On success: build validation_complete event from result; reduce + perform effects
       - DRAFTING_ACTIVITY → transition to WAITING_FOR_ACTIVITY_REVIEW; emit 'activity.draft_ready' audit event
         (actual draft generation is Phase 7; Phase 5 completes at WAITING_FOR_ACTIVITY_REVIEW)
       - Terminal phases (COMPLETED, FAILED, ABORTED): return state unchanged

    Side-effect performer (performSideEffects(effects, db)):
    - createPendingApproval: call store/audit.ts createPendingApproval(runId, proposal fields)
    - appendAuditEvent: call store/audit.ts appendAuditEvent(runId, auditType, actor, payload)
    - appendObservation: call store/audit.ts appendObservation(runId, source, redactSecrets(content))
    - emitEvent: call runEventBus.emit(runId, eventType, payload) — import from events/run-event-bus.ts
    - updateRunPhase: call store/runs.ts updateRunPhase(runId, phase)
    - updateRunStatus: call store/runs.ts updateRunStatus(runId, status)

    stepCount persistence: store stepCount as a meta observation with source 'system' and
    content JSON '{"type":"stepCount","value":N}'; read it back with listObservations filtering
    for this type. Increment after each diagnostic_proposal_ready or fix_proposal_ready event.

    createInitialState(run: Run): OrchestratorState — exported helper for Phase 6 driver bootstrap.

    Anti-pattern A1 guard (ARCHITECTURE §7): advance() NEVER calls executeApprovedCommand directly
    when processing diagnostic_proposal_ready or fix_proposal_ready events. Execution only happens
    on the command_approved incoming event, using the SshExecutor from createSshExecutor() factory.

    REFACTOR: Extract agentDispatch(state, db) and performSideEffects(effects, db) as private
    helpers inside orchestrator.ts; keep reduce() and advance() as the only exports besides types.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| agent output → driver | DiagnosticProposal / FixProposal values are untrusted; all command strings pass through validateCommandAgainstPolicy before any approval row is created |
| command_result stdout/stderr → observation | SSH output is untrusted; redactSecrets() applied before appendObservation write |
| OrchestratorEvent from HTTP caller (Phase 6) → driver | Caller provides approvalId and finalCommand; driver re-validates finalCommand through safety gate before executing (defence-in-depth) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-13 | Elevation of Privilege | advance() bypassing safety gate on command_approved | mitigate | driver re-runs validateCommandAgainstPolicy(finalCommand) on every command_approved event before calling executeApprovedCommand; mismatch → BLOCKED audit event, no execution |
| T-05-14 | Information Disclosure | Raw SSH output reaching audit/model | mitigate | redactSecrets() called on stdout+stderr before appendObservation and appendAuditEvent in performSideEffects |
| T-05-15 | Tampering | Agent output containing prompt-injection in purpose/hypothesis fields | mitigate | Fields treated as opaque strings — stored verbatim in audit log, never eval'd or executed; only command field passes the deterministic safety gate |
| T-05-16 | Denial of Service | advance() throwing on agent failure, crashing caller | mitigate | All agent calls wrapped in try/catch; failure → 'agent.unavailable' audit event + return current state; advance() always resolves |
| T-05-SC | Tampering | npm installs | accept | No new packages; only existing ai, zod, store, safety, ssh-tools imports |
</threat_model>
