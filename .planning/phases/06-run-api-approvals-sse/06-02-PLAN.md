<feature>
  <name>Approval and rejection routes (API-02)</name>
  <files>backend/src/routes/approvals.ts, backend/src/tests/approvals.test.ts</files>

  <behavior>
    POST /api/runs/:runId/approvals/:approvalId/approve:
    - Body: { editedCommand?: string, reason?: string } (both optional)
    - Look up CommandApproval by approvalId; if not found → 404 { error: "approval not found" }
    - If approval.status !== "PENDING" → 409 { error: "approval already decided", status: approval.status }
    - finalCommand = body.editedCommand ?? approval.proposed_command
    - Call advance(runId, { type: 'command_approved', approvalId, finalCommand })
    - If returned state.phase === 'WAITING_FOR_APPROVAL' AND a new command.blocked audit event was just created (advance emitted command.blocked and did NOT transition to EXECUTING/OBSERVING) → return 422 { error: "command blocked by safety policy", riskLevel: "HIGH_RISK_BLOCKED" }
    - Otherwise → 200 { status: state.status, phase: state.phase, approvalId, safetyRecheck: { riskLevel: approval.risk_level, allowed: true }, result: { from latest command_results row for this approvalId } }
    - Unknown runId → 404 { error: "run not found" }

    POST /api/runs/:runId/approvals/:approvalId/reject:
    - Body: { reason: string } (required, non-empty)
    - Zod: z.object({ reason: z.string().min(1) }) → 400 { error: "reason is required" } on failure
    - Look up CommandApproval; if not found → 404; if not PENDING → 409
    - Call advance(runId, { type: 'command_rejected', reason })
    - Return 200 { status: state.status, phase: state.phase }
    - Unknown runId → 404 { error: "run not found" }

    Blocked-command detection logic for approve:
    advance() re-validates the finalCommand via validateCommandAgainstPolicy (T-05-13 in orchestrator.ts).
    When blocked, advance() calls reduce(state, { type: 'command_blocked', ... }) which keeps phase = WAITING_FOR_APPROVAL (WAITING_FOR_APPROVAL has no command_blocked transition — it stays in WAITING_FOR_APPROVAL).
    So: if state.phase is still WAITING_FOR_APPROVAL after advance() returns on a command_approved call, it was blocked. Return 422.

    safetyRecheck.riskLevel in the 200 response:
    Return approval.risk_level — the classified value already stored in the command_approvals row (parsed from the DB via CommandApprovalSchema). Do NOT hardcode any risk level string.
  </behavior>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-05 | Tampering | editedCommand in approve body | mitigate | advance() calls validateCommandAgainstPolicy on finalCommand before SSH execution (T-05-13 in orchestrator.ts); route never bypasses this — it always passes finalCommand through advance() |
| T-06-06 | Tampering | Duplicate/stale approval (replay) | mitigate | Route checks approval.status === 'PENDING' before calling advance(); already-decided approvals return 409 |
| T-06-07 | Information Disclosure | 422 response on blocked command | mitigate | 422 returns only riskLevel (HIGH_RISK_BLOCKED) and error message — no raw command text; command is already in the audit log (redacted) |
| T-06-08 | Denial of Service | Empty-reason reject floods TRIAGING | accept | Single-team local tool; no auth layer; reason enforced non-empty (z.string().min(1)) so empty reason → 400 before any state change |
| T-06-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages in this plan |
