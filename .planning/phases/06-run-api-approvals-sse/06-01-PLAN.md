<objective>
Implement and test the four core run lifecycle routes: create, get, advance (next), and abort. These routes are thin wrappers over the existing advance() orchestrator — no state transition logic belongs here.

IMPORTANT — POST / contract (per D-02 / PRD §9): POST /api/runs must NOT call advance(). The correct sequence is: createRun() (phase=CREATED) → transition run to LOADED_CONTEXT synchronously (either via a single advance step that stops at LOADED_CONTEXT, or by storing LOADED_CONTEXT directly) → return 201 with status: "LOADED_CONTEXT". The browser then drives the run forward by calling POST /api/runs/:runId/next. Calling advance() from POST / and letting it recurse all the way to WAITING_FOR_APPROVAL would violate the PRD §9 201-response contract.

Purpose: TDD enforces the PRD §9 response shapes before implementation, preventing shape drift that would break the frontend in Phase 8.
Output: runsRouter in backend/src/routes/runs.ts with green Vitest contract tests.
</objective>

  <behavior>
    POST /api/runs:
    - Valid body { ticketId: 1 } → 201 { runId: string, status: "LOADED_CONTEXT", ticket: { id, title, priority, status, customer_name }, customerSystem: { ip, port, username, os } }
    - status in the 201 response is ALWAYS the string "LOADED_CONTEXT" — not the raw DB phase value if different
    - POST / does NOT invoke advance(); it calls createRun() then transitions the run to LOADED_CONTEXT (see implementation note)
    - Missing ticketId → 400 { error: "invalid request body" }
    - Unknown ticketId (no customer system in mock) → 404 { error: "ticket or customer system not found" }

    GET /api/runs/:runId:
    - Known runId → 200 { runId, status, phase, ticket?, timeline: AuditEvent[], pendingApproval: CommandApproval | null, activityDraft: ActivityDraft | null }
    - Unknown runId → 404 { error: "run not found" }

    POST /api/runs/:runId/next:
    - Advances orchestrator via advance(runId) — no body required
    - Returns 200 { status, phase, pendingApproval: CommandApproval | null }
    - Unknown runId → 404 { error: "run not found" }

    POST /api/runs/:runId/abort:
    - Calls advance(runId, { type: 'abort' })
    - Returns 200 { status: "ABORTED" }
    - Non-abortable phase (COMPLETED/FAILED) → returns 200 with current status (orchestrator no-ops on terminal phases)
    - Unknown runId → 404 { error: "run not found" }
  </behavior>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP body → POST /api/runs | ticketId is untrusted — validated with Zod before any store write |
| Route → orchestrator | runId from URL path — looked up with getRunById before advance(); unknown ID → 404, never passed to advance() blindly |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering | POST /api/runs body (ticketId) | mitigate | Zod safeParse z.number().int().positive() before any store write; returns 400 on failure |
| T-06-02 | Information Disclosure | GET /api/runs/:runId timeline | mitigate | getAuditEvents reads already-redacted rows (redactSecrets called at appendAuditEvent write time in audit.ts); no raw secrets reach the response |
| T-06-03 | Elevation of Privilege | POST /:runId/abort on terminal phase | accept | orchestrator reducer is a no-op on COMPLETED/FAILED/ABORTED; abort on terminal phase returns current state harmlessly |
| T-06-04 | Spoofing | SSRF via ticketId → Phoenix client | mitigate | Phoenix client is a typed wrapper with a fixed base URL from env.ts; ticketId is an integer — no URL injection surface |
| T-06-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages; slopcheck not needed |
